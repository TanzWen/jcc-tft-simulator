本文件是 JCC 项目中 Claude 及其他协作智能体必须遵守的项目级约定。若与用户当前指令冲突，以用户当前明确指令为准。

## 一、全局协作规则

### 修改前先确认方案

- 任何涉及修改代码、配置、文档、数据库或系统状态的操作，必须先简要说明拟改动内容、影响范围和验证方式，得到用户确认后再执行。
- 阅读文件、搜索内容以及执行 `git status`、`git log`、`git diff` 等只读操作，无需事先确认。
- 用户明确说“直接改”“不用问”“一步到位”等同义指令时，可跳过当次确认。
- 用户确认的范围是本次修改边界；如过程中发现需要明显扩大范围，应再次说明并确认。

### Git

- 未经用户明确要求，不主动创建 commit、推送远端、建立分支或发起 PR。
- 创建 commit 时，message 中不得添加 `Co-Authored-By: Claude ...`，也不得添加其他 Claude 共作署名。
- 不覆盖或清理来源不明的工作区改动；修改前先检查 `git status`，只处理本任务相关文件。
- 每次 commit 前必须检查并同步更新 `CLAUDE.md` 与 `README.md`，确保技术栈、目录、命令、数据库结构和功能状态与代码一致。
- commit 前至少运行与改动风险相匹配的测试，并检查待提交 diff。

### 新建项目脚手架

创建新项目时，应自动完成：

1. 按技术栈创建合适的 `.gitignore`。
2. 创建项目级 `CLAUDE.md`，说明技术栈、目录约定、运行方式与协作注意点。
3. 创建 `README.md`，至少包含项目简介、安装、运行和基本用法。
4. 创建 `docs/` 存放方案文档，并将 `docs/` 加入 `.gitignore`，使方案文档默认不进入版本控制。

## 二、项目技术约定

### 技术栈

- Python 3.11+，统一使用 Conda base 环境中的 `python` 命令，不使用 `python3` 命令。
- 运行时只使用 Python 标准库；新增第三方依赖前必须单独说明原因并取得确认。
- 数据库存储使用 SQLite，默认文件为 `data/jcc.db`。
- 前端使用原生 HTML、CSS、JavaScript，无框架、无构建步骤、无前端 CDN 依赖。
- 本地 Web 服务使用 `http.server`，不额外引入 Web 框架。

已实现的数据能力：

- 从官网 `basicConfig.js` 识别当前 mode 和 season，从官方版本索引选择当前日期已经生效的最新版本。
- 获取英雄、技能、各星级属性、羁绊档位、装备、合成组件和强化符文。
- 按 `mode + season + version` 保存原始快照、来源 URL 和 SHA-256。
- 将结构化数据事务化写入 SQLite，并为旧数据库补充新增字段。
- 下载并校验英雄头像与官网英雄页的 1624×750 横版原画，已存在资源可复用。
- 对网络断流和不完整响应执行重试；解析或资源校验失败时停止入库。

已实现的阵容模拟器能力：

- 液态玻璃单页界面，左侧包含英雄池和装备库。
- 英雄支持费用、羁绊和关键词组合筛选；装备支持类型及文本筛选。
- 阵容支持 1–12 人口、每名英雄最多 3 件装备、转职纹章追加羁绊。
- 支持 4×7 交错六边形棋盘、拖放站位、英雄互换及下场。
- 实时统计羁绊人数和激活档位，并展示装备基础属性汇总。
- 推荐补强会寻找可推进羁绊档位的英雄。
- 阵容、配装和站位通过 URL hash 保存与分享，分别使用 `h`、`i`、`p` 参数。
- `GET /api/season` 返回当前赛季英雄、羁绊、装备及其关联关系；本地图片通过 `/assets/` 提供。

当前尚未完成：

- 真实对局数据采集及胜率、前四率、登顶率等实战指标。
- 技能、装备和强化符文自然语言效果的完整结构化。
- 可解释的理论强度评分器、战斗模拟器和阵容自动生成器。
- 赛季差异报告、定时更新、日志和数据质量报告。

因此当前数据足以计算基础属性、羁绊完整度、费用曲线、阵容结构和装备适配等理论指标，但不得把这些指标描述为真实胜率或版本强度排名。

### 目录约定

```text
src/jcc/
├── collector.py       # 官网版本识别、结构化数据采集、清洗和原始快照
├── database.py        # SQLite schema、迁移与事务入库
├── images.py          # 英雄头像和 1624×750 横版原画下载与校验
├── cli.py             # 数据更新命令
└── web/
    ├── server.py      # 本地 HTTP 服务、数据 API、本地图片服务
    └── static/        # 原生前端页面
tests/                 # unittest 离线测试
data/raw/              # 按赛季版本保存的官网原始响应，不提交
data/assets/           # 本地头像与横版原画，不提交
data/jcc.db            # SQLite 数据库，不提交
```

### 数据采集

- 数据源只使用金铲铲之战官网及其腾讯官方 CDN 公开资源。
- 采集必须遵守官网使用条款、Robots 协议和合理访问频率，不绕过登录、验证码或访问限制。
- 不在代码中硬编码当前赛季版本；通过官网 `basicConfig.js` 和版本索引动态识别。
- 官网提前发布但尚未到生效日期的数据不得自动选为当前版本。
- 每次采集须保留原始响应和来源 URL，并记录内容哈希，确保结果可追溯。
- 网络断流、不完整响应、非法 JSON 或非法图片必须重试或明确失败，不得将半份数据写入数据库。
- 重复运行同一赛季版本应保持幂等，不产生重复记录。
- 更新赛季数据只能通过显式运行 `python -m jcc.cli`；`start.sh` 仅在数据库缺失或缺字段时兜底采集，不得让它承担版本追新职责。
- 官网字段结构变化时，优先依据 `data/raw/` 下的快照定位，再调整 `collector.py` 中对应的 `transform_*` 函数。

### 数据库

- 所有业务数据必须通过 `season_id` 或 `mode + season + version` 隔离，禁止混用不同赛季数据。
- 当前核心表包括 `seasons`、`heroes`、`traits`、`hero_traits`、`equipment`、`augments`、`compositions`。
- `compositions` 是用户数据，不由采集覆盖：采集只重建当前赛季的资料表，存档随 `season_id` 外键级联，不得在采集流程里删除。
- 数据表结构改动集中在 `database.py` 的 `SCHEMA`，同时提供对已有数据库安全升级的迁移逻辑。
- 修改字段后必须同步检查：采集转换、INSERT/UPDATE、Web API 查询、测试、README 数据库说明和启动脚本 schema 检查。
- 入库使用事务；解析结果为空或资源下载不完整时，不覆盖现有完整数据。
- 数据库改动后运行 `PRAGMA foreign_key_check` 与 `PRAGMA integrity_check`。

### 图片资源

- 每位英雄同时保留官网头像和英雄页使用的 1624×750 横版原画。
- 头像保存于 `data/assets/heroes/<赛季版本>/`，横版原画保存于其 `full/` 子目录。
- 图片地址必须从官网字段或 `hero_pic_big` 配置动态生成，不手写特定赛季路径。
- 下载后校验图片文件头；临时文件使用 `.part` 后缀，完整下载后再原子替换。
- 数据库分别保存官网地址与本地相对路径；前端优先使用本地横版原画，头像用于棋盘等小尺寸场景。
- 图片裁切只使用 CSS，例如 `object-fit: cover`，不在 JavaScript 中计算尺寸或生成派生缩略图。

### 前端与阵容逻辑

- 页面数据通过本地 API 读取，不在前端硬编码英雄、羁绊或赛季数据。
- 前端只保存设备侧交互状态；正在编辑的阵容状态使用 URL hash，不写入数据库。
- 用户主动保存的阵容存档写入 `compositions` 表，通过 `/api/compositions` 读写；服务端负责全部校验（赛季归属、装备数量、站位范围、重名），前端不得假设自己送来的数据一定合法。
- 同名英雄在羁绊计数中只计算一次。
- 羁绊档位以 `traits.levels_json` 的 `count` 和 `color` 为准。
- 涉及“强度”的功能必须区分理论评分与真实胜率；没有对局统计时不得把启发式评分表述为实战胜率。
- 动态文本写入 HTML 时注意转义，新增交互需兼顾键盘操作、移动端和空数据状态。

## 三、开发与验证

常用命令：

```bash
./start.sh                                             # 一键启动，必要时自动采集
PYTHONPATH=src python -m jcc.cli                       # 更新数据与图片
PYTHONPATH=src python -m jcc.cli --skip-images         # 更新数据，不下载缺失图片
PYTHONPATH=src python -m jcc.web.server --open         # 启动本地页面
PYTHONPATH=src python -m unittest discover -s tests -v # 运行测试
```

最低验证要求：

- 纯转换逻辑：运行相关 unittest。
- 采集器改动：运行全部 unittest，并对真实官网数据至少完成一次端到端验证。
- 数据库改动：验证记录数、外键、完整性和重复执行结果。
- 图片改动：验证文件数量、文件头、数据库路径及本地 API 地址。
- 前端改动：至少验证服务可启动、API 可返回数据、静态资源路径有效；如用户要求视觉检查，再进行浏览器 QA。

`start.sh` 的数据库字段检查必须与当前 schema 同步；新增必要字段时，应同步更新脚本判断条件。

当前已知维护项：

- `start.sh` 仍按 `python3.14` 至 `python3` 的顺序自动寻找解释器，没有优先使用 Conda base 的 `python`，与当前命令规范不完全一致。
- `start.sh` 会检查头像、横版原画、装备和强化符文所需结构；服务端读取赛季前也会自动执行可重复的 schema 迁移。
- 修复上述启动脚本问题属于代码修改，必须另行说明方案并取得用户确认。
